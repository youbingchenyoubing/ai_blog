# Superpowers 最佳实践

## 1. 设计原则

### 1.1 简单性优先
- 选择最简单的可行方案
- 删除不必要的功能
- 避免过度工程化
- 每个功能必须有明确的价值

### 1.2 YAGNI（你不会需要它）
- 不要为未来可能需要的功能写代码
- 只实现当前明确需要的功能
- 当需求变化时再重构

### 1.3 DRY（不要重复自己）
- 重复代码是 bug 的温床
- 提取公共逻辑到函数/类
- 但不要过早抽象

### 1.4 KISS（保持简单）
- 代码应该易于理解
- 避免复杂的逻辑链
- 每个函数只做一件事

## 2. 测试策略

### 2.1 测试金字塔
- 单元测试：70-80% 覆盖率
- 集成测试：20-25% 覆盖率  
- E2E 测试：5-10% 覆盖率

### 2.2 测试命名规范
- 描述性：`test_user_cannot_login_with_wrong_password`
- Given-When-Then 模式：
  - Given：前提条件
  - When：执行操作
  - Then：预期结果

### 2.3 测试反模式
- ❌ 测试实现细节而非行为
- ❌ 测试过于脆弱（依赖具体实现）
- ❌ 测试没有实际价值
- ❌ 测试不独立（有依赖）

## 3. 代码质量

### 3.1 函数设计
- 函数长度 ≤ 20 行
- 参数数量 ≤ 3 个
- 有明确的返回值
- 有清晰的错误处理

### 3.2 命名规范
- 变量：描述性名词 `user_count`
- 函数：动词+名词 `calculate_total`
- 布尔值：`is_`、`has_`、`can_` 前缀
- 常量：全大写 `MAX_USERS`

### 3.3 注释原则
- 解释"为什么"，而不是"是什么"
- 复杂的业务逻辑需要注释
- API 文档必须完整
- 过时的注释比没有注释更糟

## 4. 子代理管理

### 4.1 任务分配
- 每个子代理处理单一职责
- 任务之间无依赖
- 提供完整的上下文
- 设置明确的完成标准

### 4.2 监控与恢复
- 定期检查子代理进度
- 超时自动终止
- 偏离计划时重新派发
- 记录所有子代理活动

### 4.3 结果验证
- 验证输出是否符合预期
- 检查代码质量
- 运行自动化测试
- 生成验证报告

## 5. 错误处理

### 5.1 防御性编程
- 验证所有输入
- 处理边界情况
- 提供有意义的错误信息
- 记录错误上下文

### 5.2 恢复策略
- 重试机制（带指数退避）
- 降级方案
- 优雅失败
- 用户友好的错误提示

## 6. 文档标准

### 6.1 设计文档
- 问题陈述清晰
- 方案对比完整
- 技术决策有依据
- 风险识别全面

### 6.2 API 文档
- 端点描述
- 请求/响应示例
- 错误码说明
- 认证要求

### 6.3 部署文档
- 环境要求
- 安装步骤
- 配置说明
- 故障排除

## 7. 性能考虑

### 7.1 响应时间
- API 响应 < 200ms
- 页面加载 < 2s
- 数据库查询 < 100ms

### 7.2 资源使用
- 内存使用有上限
- 连接数有限制
- 文件大小有控制
- 缓存策略合理

## 8. 安全最佳实践

### 8.1 输入验证
- 所有输入都要验证
- 使用白名单而非黑名单
- 转义特殊字符
- 限制输入长度

### 8.2 认证授权
- 使用标准认证协议
- 最小权限原则
- 定期轮换密钥
- 记录所有访问

### 8.3 数据保护
- 敏感数据加密
- 传输使用 TLS
- 定期备份
- 访问日志完整
