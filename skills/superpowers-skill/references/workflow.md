# Superpowers 工作流详细指南

## 1. Brainstorming（需求澄清）

### 核心原则
- 永远不要直接跳到写代码
- 先问"你真正想实现什么？"
- 探索至少 3 种方案
- 设计文档每块不超过 200 字

### 执行流程
1. 理解用户表面需求
2. 追问深层目标："这个功能最终要解决什么问题？"
3. 提出 3+ 替代方案，分析各方案优劣
4. 推荐最优方案并解释原因
5. 生成设计文档 `design.md`，分块展示
6. 逐块确认用户理解并签字

### 设计文档模板
```markdown
# 项目名称：设计文档

## 1. 问题陈述
[一句话描述要解决的问题]

## 2. 目标用户
[谁会使用这个功能？]

## 3. 设计方案
### 方案 A：[名称]
- 优点：
- 缺点：
- 复杂度：

### 方案 B：[名称]
- 优点：
- 缺点：
- 复杂度：

### 推荐方案：[名称]
- 理由：

## 4. 技术架构
- 前端：
- 后端：
- 数据库：
- API 设计：

## 5. 关键风险
- [风险 1]
- [风险 2]

## 6. 成功标准
- [标准 1]
- [标准 2]
```

## 2. Implementation Planning（实现计划）

### 任务分解原则
- 每个任务 2-5 分钟完成
- 必须是可独立验证的原子单元
- 包含精确文件路径
- 包含完整验证步骤
- 遵循 YAGNI 和 DRY

### 计划文档模板
```markdown
# 实现计划

## 任务列表

### 任务 1：[名称]
- 文件：[完整路径]
- 描述：[具体要实现什么]
- 验证：[如何确认完成]
- 依赖：[前置任务]
- 预计时间：[分钟]

### 任务 2：[名称]
...
```

## 3. TDD（测试驱动开发）

### 红-绿-重构循环
1. **RED**：写一个失败的测试
2. **GREEN**：写最少代码让测试通过
3. **REFACTOR**：重构代码保持整洁
4. **COMMIT**：提交代码

### 关键规则
- 永远不写没有对应测试的代码
- 先跑测试看它失败，再写实现代码
- 删除任何在测试前写的实现代码
- 每次提交只包含一个循环

## 4. Subagent-Driven Development（子代理驱动开发）

### 两阶段审查
- **第一阶段：规范符合性**
  - 是否严格遵循计划？
  - 文件路径是否正确？
  - API 签名是否匹配？
  
- **第二阶段：代码质量**
  - 是否遵循编码规范？
  - 是否有足够测试？
  - 是否有性能问题？

### 批量执行规则
- 3-5 个任务一批
- 每批完成后设置人工检查点
- 子代理偏离计划时立即停止

## 5. Code Review（代码审查）

### 问题分级
- **关键**：安全漏洞、数据丢失风险、核心功能缺失 → 阻塞
- **主要**：性能问题、不正确的错误处理、可维护性问题
- **次要**：命名建议、注释改进、代码风格

### 审查清单
- [ ] 代码是否符合设计文档？
- [ ] 是否遵循实现计划？
- [ ] 是否有完整测试覆盖？
- [ ] 是否处理了错误情况？
- [ ] 是否有明显的性能问题？
- [ ] 是否有硬编码或魔法数字？

## 6. Finishing（完成分支）

### 完成检查清单
- [ ] 所有测试通过
- [ ] 无未处理的 lint 错误
- [ ] 文档已更新
- [ ] 代码审查已完成

### 选项
1. **合并**：合并到主分支
2. **创建 PR**：创建拉取请求等待审查
3. **保留分支**：保留工作分支
4. **丢弃**：放弃所有更改
