## 一、Loop Engineering 是什么

2026 年 6 月，一个概念在一周内被三位顶级工程师同时点燃：

- **Peter Steinberger**（OpenClaw 作者）：别再给 Coding Agent 逐行写提示了，要设计让 Agent 自己循环运转的系统；
- **Boris Cherny**（Claude Code 负责人）：他不再手动提示 Claude，而是写 Loop 让 Claude 自己运行；
- **Addy Osmani**（Google Chrome 团队工程师）：在博客上发表长文，正式命名这个概念——**Loop Engineering**。

三人的共识指向同一个动作：**设计的对象从"Agent 的单次行为"转变为"驱动 Agent 的完整系统"。**

Loop Engineering 的精确定义：

> **替换你自己作为给 Agent 下指令的人，转而去设计一个能自动完成这件事的系统。** 你不再是引擎，而是设计引擎的人。

用一句话说：**把 Loop 的执行者从"你"换成"Agent"。**

原本由你亲自跑的"发现问题→想办法→动手→检查"循环，现在交给 Agent 去跑。你要做的，只剩开头**把目标（验收标准）说清楚**。

> 💡 **Self-prompting——循环的关键拼图**：上一轮跑完后，不由人来想"下一步该问什么"，而是让 Agent 根据已有进展，自己写下一轮要执行的 Prompt。这一步看似不起眼，却是把"人当循环体"换成"Agent 当循环体"的关键机制。人不再负责接力，循环自己续上了。

## 二、从手工循环到自动循环

### 手工循环：你就是循环的执行者

回想过去几个月，与 Coding Agent 打交道的节奏大概是这样：

```
你写 Prompt → Agent 干活 → 产出结果 → 你读结果 → 判断哪里不对 → 再写 Prompt …… → 直到做完
```

这套节奏熟练到让人忽略了一个事实：**它本身就是一个完整的循环**。

发现问题、想办法、动手、检查结果，每个环节都在，循环也确实在转。**区别只在于：循环体的执行者是你自己。**

Agent 干完一轮就停下，安静等着。真正负责"判断、决定、再来一轮"的角色，由人来扮演。**这一轮里，是机器在等你。**

代价就藏在那条从"你读结果"回到"你写 Prompt"的折返线里：

- Agent 每跑完一轮，你都得回到屏幕前查看，否则循环就**停在那儿不动**
- 注意力被反复切走又切回，刚开始写另一份文档，那边一声"完成"提示音，又得切回来
- 一天下来，**注意力被切得很碎，上下文在几件事间反复搬运**，真正连续投入的时间反而不多

不知你是否也有同感——用 AI Coding 之后，类似 ADHD 的状态反而更明显了？

社区里甚至出现过用软件通知、ESP32 硬件来提醒"Agent 已经跑完了"的小工具，从侧面也可以印证"人盯着循环"这件事有多普遍。

当你把代价摆在手工循环旁边，问题就清楚了：**循环本身没问题，问题在于让人去当那个一刻不能离开的循环体。**

### 自动循环：人退到设目标的位置

Loop Engineering 的本质：**把循环体的执行者从"你"换成"Agent"。**

手工循环里那个一刻不能离开的位置，现在让 Agent 顶上。你要做的，是从"每一轮都判断、都接力"**缩减成"开头把目标说清楚一次"**。

这个新循环并不神秘，就是把你脑子里那几步，显式地交给 Agent 去走：

```
用户设定目标 → Discovery（发现）→ Plan（规划）→ Execute（执行）→ Verify（验证）
     ↑                                                              │
     │                    达标 → Ship                               │
     │                    不达标 → Iterate（Self-prompting 再来一轮）←┘
                          │
                     Memory（对话之外的持久记忆）
```

**变的不是循环本身，是谁来当那个一刻不能离开的执行者。**

## 三、四层架构栈

Loop Engineering 并非取代前三层，而是在它们之上叠加了一层。AI 工程实践可以划分为四层栈：

| 层级 | 关注对象 | 核心问题 |
|------|----------|----------|
| **Prompt Engineering** | 写好一条提示 | 我该告诉模型什么？ |
| **Context Engineering** | 窗口里放什么 | 该检索什么、总结什么、清除什么？ |
| **Harness Engineering** | 武装单次运行 | 哪些工具、哪些动作、怎样算完成？ |
| **Loop Engineering** | 在 Harness 上调度 | 如何让它一遍又一遍地自动运行？ |

每一层向上，关注的单元就放大一号：从一句话，到一个窗口，到一次运行，最终到一个能自我运转的循环。

Loop Engineering 的核心新增能力是三个动词：

- **定时运行（Runs on a timer）**：Agent 自己醒来开工，而不是等你点一下
- **派生助手（Spawns helpers）**：一个 Orchestrator 把任务拆开，同时派给多个 Agent 并行去做
- **自我喂养（Feeds itself）**：上一轮的进展自动成为下一轮的输入，循环自己续上

> 💡 **Fan-out / Fan-in**：Fan-out 是一个 Orchestrator 把任务拆开，同时派给多个 Agent 并行去做；Fan-in 是这些 Agent 各自跑完后，结果再汇聚回来，由一处统一收口、汇总。**散出去是为了快，收回来是为了不乱。**

## 四、五步动作与六大组件

### 五步动作

一个 Loop 的"一圈"不是空转，而是完成五个具体动作：

| 动作 | 作用 | 晨间巡检 Loop 示例 |
|------|------|---------------------|
| **Discovery（发现）** | 自主找出本轮该做什么 | Skill 读取 CI 失败、Open Issues、近期 Commit |
| **Handoff（交接）** | 将任务隔离地交给执行 Agent | 每个发现打开独立的 git worktree |
| **Verification（验证）** | 用另一个 Agent 说"不" | 第二个子 Agent 对照测试和项目规范审查 |
| **Persistence（持久化）** | 将结果写到对话之外 | 提交 PR、更新看板、写入状态文件 |
| **Scheduling（调度）** | 让循环能自动转起来 | 每天早上自动触发，状态文件让次日接续 |

### 六大组件

如果说"动作"是 Loop 的"行为"，那么"组件"就是它的"部件"：

| 组件 | 本质 | 对应动作 | 关键说明 |
|------|------|----------|----------|
| **Automations（自动化）** | 定时或触发式调度器 | Scheduling | 让循环按时起床，不用人手按 |
| **Worktrees（工作树）** | 并行 Agent 的隔离目录 | Handoff | **不隔离 = 并行 Agent 改同一目录 = 合并一团糟（Tangled Loop 的前因）** |
| **Skills（技能）** | 永久化的项目知识（SKILL.md） | Discovery | 项目规范、编码风格、常见踩坑，写进文件随时取用 |
| **Connectors（连接器）** | 基于 **MCP 协议**的外部系统挂钩 | Persistence / Discovery | Agent 访问 GitHub / Sentry / Stripe 等外部系统的统一接口 |
| **Sub-agents（子代理）** | 写代码的与审代码的分离 | Verification | Generator 负责产出，Evaluator 负责质疑，角色隔离 |
| **Memory（记忆）** | 磁盘上的持久状态 | Persistence | **活在对话之外，循环才稳得住** |

> 💡 **Memory：活在对话之外**。一轮循环跑完，进展得留下来，下一轮才接得上。常见误区是把所有进展都堆回 Context Window，指望 Agent"记得"上一轮干了什么——但工作记忆有限，容易因 attention 流失而被"遗忘"。正确做法是把进度记录在对话之外：写进文件、写进一份结构化的状态记录，让它独立于任何一次对话而存在。

## 五、Generator / Evaluator 分离

### Agent 不能给自己打分

让 Agent 评判自己刚写的代码，它会自信地给出好评——即使质量平庸。这不是智商问题，而是**"自己批自己作业"的结构性缺陷**：Agent 的上下文里塞满了"为什么这样写"的自我说服链条，它看到的不是结果，而是导致结果的理由。

### 用另一个 Agent 说"不"

让写代码的 Agent 变得更自我批评，效果很差。但**单独调校一个 Evaluator Agent，让它默认怀疑、通过行动验证（而非阅读判断）、并由一个全新的模型执行最终裁决**——这远比让 Generator 自我批判更可行。

Claude Code 的 `/goal` 原语就是这一思想的产物：给 Agent 一个停止条件，让它循环运行直到满足条件，而**判断是否满足条件的是一个更快、更小的独立模型**，而非正在干活的那个。

> 💡 **grill-me 机制**：设定目标时，你未必一开始就能把验收标准想清楚。grill-me 的做法是让 AI 反向采访你——它提问、你回答、它总结——最终把模糊的意图收敛成一份清晰的目标和验收标准。这是 `/goal` 设定的辅助手段，也是人留在"掌舵位置"的具体体现。

## 六、Open Loop vs Closed Loop

把循环体交给 Agent 之后，最先要面对的不是"它能不能干"，而是"**它要花多少**"。同样一句目标，落到不同形态的 Loop 上，开销能差出一个量级。

| 维度 | Open Loop | Closed Loop |
|------|-----------|-------------|
| **起点** | 轻入口，一句话即可 | 有界目标，路径大致看得清 |
| **探索面** | 广，常发现事先没想到的方向 | 窄，基本不给你意外惊喜 |
| **开销** | 难预估，只能靠预算上限兜住 | 可控，能估出大概几轮收敛 |
| **验收** | 终点模糊，何时停由 Agent 自己回答 | 客观可判定，达标即停 |
| **适用场景** | 预算充裕、以探索为目的 | 交付场景、在可预期开销内做完确定的事 |
| **代价** | Token 消耗难以事前预估 | 不会给你惊喜 |

### Open Loop 案例：Crewlet 管理创业公司经营数据

Crewlet 把这些数据源全部交给了一个类似中央处理器的 OpenClaw Agent：GitHub 代码仓库、Sentry 日志、Google Analytics 埋点与行为数据、Supabase 业务主库、AB 实验数据、Stripe 支付、X / Discord 社交数据……

这个 Agent 每隔一段时间"醒来"，自主查看核心经营数据与近期错误日志，能做的事包括：

- **发现经营指标问题并归因**
- **从错误日志中定位 Bug**，直接改 GitHub 代码并提交 PR 上线
- 发现大量用户卡在付费这一步，原因是 Max / Pro 账号页面说明不清
- 给即将到期的用户发送挽回邮件
- 改 Agent 代码、增强 Prompt Caching，提性能、省开支
- 发起 AB 实验并分析报告，进而迭代自身功能

效果出奇地好。代价是开销难以预估。

### Closed Loop 案例：为工具库补测试

验收标准天然客观——这三条都能由机器给出 0/1 判定：

- acceptance test 全过
- typecheck 通过
- 不破坏现有测试

**"何时停"被一句 `npm test` 全绿替代了。** 循环有了必须收敛的客观靶子，轮数和开销也就有了大致上界。

### 作者建议

**多数情况先从 Closed Loop 开始。** 先把一件目标清楚的事用 Closed Loop 跑顺，摸清 Agent 在你这套流程里的脾气，再按需把边界一点点放开。反过来，一上来就 Open Loop，往往开销和结果都不容易把控。

## 七、五种失败模式

Loop 的失败不是随机的，而是**五个动作中跳掉一个的必然结果**：

| 跳过的动作 | 失败模式 | 症状 |
|------------|----------|------|
| **Verification** | **Nodding Loop（点头循环）** | 运行几百轮从未说过一次"不"，自我批准垃圾代码 |
| **Persistence** | **Amnesiac Loop（失忆循环）** | 每天从零开始，重复发现、重复冲突，无累积进展 |
| **Scheduling** | **Manual Loop（手动循环）** | 演示当天跑得漂亮，演示完再没跑过 |
| **Discovery** | **Blind Loop（盲眼循环）** | 人类每天早上还在决定 Loop 该做什么 |
| **Handoff** | **Tangled Loop（纠缠循环）** | 并行 Agent 改同一个目录，合并时一团糟 |

记住这张对照表：**每种失败模式，恰是对应动作被跳过的直接后果。** 把五个动作都补上，Loop 就能稳住。

## 八、四种隐性成本

Loop 在欢快地自动运转时，也在静默地积累四种成本——而且它们会**互相加强**：

```
┌─────────────────┐     Loop写的代码越多     ┌─────────────────┐
│ Verification    │ ──────────────────────→ │ Comprehension   │
│ Debt（验证债务） │     人越读不过来         │ Rot（理解腐烂）  │
└────────┬────────┘                          └────────┬────────┘
         ↑                                            │
   Token越烧越多                                      │人越懒得看
         │                                            ↓
┌────────┴────────┐    人越不想干预     ┌─────────────────┐
│ Token Blowout   │ ←────────────────  │ Cognitive       │
│（Token爆炸）     │                    │ Surrender       │
│                 │                    │（认知投降）      │
└─────────────────┘                    └─────────────────┘
```

1. **Verification Debt（验证债务）**：每个合并的 PR 都节省了时间，但未验证的输出在测试覆盖不到的缝隙中堆积，直到某个发布日早晨集中爆发。
2. **Comprehension Rot（理解腐烂）**：Loop 写的代码越多，人类对代码库的真实理解越滞后。读代码比写代码无聊，而 Loop 已经替你写完了。
3. **Cognitive Surrender（认知投降）**：Loop 越可靠，人类越懒得发表意见，最终从"没时间看"变成"懒得看"。
4. **Token Blowout（Token 爆炸）**：Loop 会派生助手、重试、无限循环，一个 Bug 可能烧掉整晚的预算。

**这四种成本形成正反馈环**：验证债务→理解腐烂→认知投降→不再设上限→Token爆炸→更多未验证产出→验证债务更深。打破这个环的关键，就是下一节的三条纪律。

## 九、实践场景详解

### 场景 1：为工具库补测试（Closed Loop 典范）

这是一个研发场景：给一个工具库补测试、顺手修掉暴露出来的 bug。

它的好处是**验收标准天然客观**——三条都能由机器给出 0/1 判定：

- 一组 acceptance test 全过
- typecheck 通过
- 不破坏现有测试

把循环工程的全流程放进这个任务，每一步都落得很实：

```
目标（/goal）→ Discovery（扫描仓库，发现缺失测试的模块）
            → Plan（列出要补的测试清单）
            → Execute（写测试、修bug，可 fan-out 并行）
            → Verify（跑 npm test + typecheck）
            → 达标 → Ship
            → 不达标 → Self-prompting 回到 Discovery 再来一轮
                ↑
            Memory：每轮进展写进状态文件，下一轮接续
```

**循环 Prompt 骨架**：

```
/goal 目标：为 <工具库> 补齐测试并修复其中的 bug。
验收标准：
  1. acceptance test 全过
  2. typecheck 通过
  3. 不破坏现有测试
循环条件：只要测试不全绿就继续修，全绿才停。
```

> 💡 **Closed Loop 为什么能成立**？关键就一条：验收是客观的。测试、typecheck、benchmark 给出的是 0/1 信号——要么全绿，要么有红，没有中间地带。Agent 不需要去猜"我做完了没""够不够好"，那个判断被外包给了一套确定性检查。Open Loop 开销难估，根源在于"何时停"由 Agent 自己回答。而这里，"何时停"被一句 `npm test` 全绿替代了。

实践中，这个"把验收结果塞回循环"的动作可以自动化：用 hook 或 CLI 命令的退出码来驱动。比如约定退出码非 0 即失败，配合 Claude Code 的 PostToolUse 钩子，让每次测试结果自动回流到下一轮，省去人工搬运。

### 场景 2：网店 GMV 增长（Orchestrator + Sub-Agent 并行）

一个想自我增长的电商小店，目标是"让网店 GMV 持续增长"。

这一轮，主角换成了一个 **Orchestrator（编排）**。它掌管这个开放目标，但真正做的事很具体：

开工前，先读上一轮留下的 `next_steps`——那是一份记忆，记着上次没做完、值得这轮接着做的事。读完才决定，这轮派谁去干什么。

Orchestrator 派出三个 Sub-Agent，并行工作、各管一摊：

| Sub-Agent | 职责 | 产出 |
|-----------|------|------|
| **Builder** | 构建引流磁铁（Lead Magnet） | 一个小测验，用免费内容换取访客联系方式 |
| **Scout** | 扫描竞品和趋势 | 选题排序清单，标注哪些竞品已做过 |
| **Growth** | 优化现有页面 | 落地页改进动作列表 |

三个 Sub-Agent 各自跑完，产出还是散的。**汇总与记忆**由 Orchestrator 收口：

- 把 Builder 的 quiz、Scout 的选题排序、Growth 的落地动作，合并成一份统一的行动计划
- 写回 `next_steps`——这份文件就是这套循环的**跨周期记忆**

下一轮开工，Orchestrator 第一件事就是再读它。**于是这轮的尾，接上了下轮的头。**

每轮结束时，Orchestrator 用几条简单条件判断要不要继续：

- 站点这周是否还在增长？
- Scout 手里是否还压着 ≥3 条未消化的新选题？
- 下一个 lead magnet 是否已经定下来？

条件没满足就接着转；都满足了就暂停，等下个周期再来。

> 💡 **Lead Magnet（引流磁铁）**：一份免费、对用户有点用的小东西。一份清单、一个测算工具、一个小测验，用它换取访客的联系方式（通常是邮箱）。它不直接卖货，而是先把陌生流量变成你能再次触达的潜在客户。

### 场景 3：晨间巡检（Osmani 的日常自动化）

Addy Osmani 的晨间巡检 Loop 每天自动触发：

- 读取昨日失败的 CI、未关闭的 Issues、近期 Commit
- 对值得处理的事项，打开独立 worktree
- 一个子 Agent 起草修复，另一个子 Agent 审查
- 自动提交 PR 并更新看板
- 无法处理的丢进收件箱，状态文件保证次日接续

### 扩展场景速写

这副骨架能套到很多别的事情上。每个都按"**触发时机 → 读什么 → 怎么处理 → 循环条件**"来看：

| 场景 | 触发 | 读什么 | 处理方式 | 循环条件 |
|------|------|--------|----------|----------|
| **自由职业者** | 每周五 | 各项目文件夹变化 | 带客户记忆起草进度更新，丢进 review 文件夹等人过目 | 每个合作客户这周都更新到位？ |
| **学生/追新的人** | 每周日 | 领域进展 | 按相关度打分，低于阈值过滤，写成简报，比对历史去重 | 本周 ≥2 条真正算新的进展？ |
| **网店老板** | 每月1号 | 销售数据 | 挑高流量低转化商品重写描述，给畅销品写推广文案 | 记录改了什么、为什么改，下月回顾效果 |
| **创作者** | 每周一 | 选题清单 + 90天内容表现 | 打分排序 top5，标出竞品已做过，避免撞车 | 本周有没有够格的新选题？ |

> 💡 **相关度阈值在拦什么？** 学生场景里"低于阈值的过滤掉"看似不起眼，却是循环能长期跑而不烦人的关键——每周新进展里大多是噪声，不设门槛，简报很快就会被无关信息塞满。阈值的作用是替你先把"勉强相关"的挡在外面，只让够格的进来。

**最后提一句**：这些循环跑得再顺，AI 也不是每一步都对。尤其像"判断哪个选题会火"这种事，它并不可靠。更稳妥的用法是，把循环产出当作**"多了一个有理有据的参考意见"**，而不是一个说了算的 Oracle。它替你把信息收齐、把初步判断摆出来；至于最后怎么取舍，仍留给自己。

### 基础设施类

| 工具 | 本质 | 如何支撑 Loop |
|------|------|---------------|
| **MCP（Model Context Protocol）** | 连接器标准协议 | Agent 访问 GitHub / Sentry / Jira / Stripe 等外部系统的统一接口，是 Connectors 组件的底层协议 |

| **GitHub Actions** | CI/CD + 定时调度 | 充当 Scheduling 组件，定时触发 Loop 运行；也充当 Verification 组件，硬编码的门控不让 Agent 跳过 |

### 跨工具链能力对照：Claude Code vs Codex

| 能力 | Claude Code | Codex CLI |
|------|-------------|-----------|
| 循环原语 | `/loop` | `--full-auto` |
| 目标设定 | `/goal` | `--goal`（等效） |
| 子 Agent | spawn sub-agent | spawn sub-agent |

| 本地文件访问 | 是 | 是（沙箱内） |

**核心洞察**：Loop Engineering 是一组能力，而非某个产品的独占功能。不同的工具可以实现同一套 Loop，选择取决于你的场景和偏好。

## 十一、三条安全纪律

### 1. 永远抽样阅读（Read a Sample, Always）

不需要读所有输出，但**每天读一个有代表性的样本，并强迫自己解释它做了什么、为什么这样做**。无法解释 = 你的心理地图已经落后于代码库。

### 2. 上线前设上限（Cap Before You Ship）

在 Loop 第一次无人值守运行前，设置：

- **单次预算上限**
- **每日预算上限**
- **最大重试次数**

这不是为了省钱，而是**把开放式风险转化为有界风险**。

### 3. 留一扇门（Keep One Door Open）

在 Loop 中至少设置一个**人类检查点**。不是为了每次都干预，而是为了**保持"能够干预"的位置**。把所有门焊死的工程师，会在需要进去的那天发现自己已经丢了钥匙。

> 💡 Stripe Minions 的做法就是典范：虽然每周合并 1300+ 个机器 PR，但**最终仍由人类 Review**。这个检查点不是为了挡住每一个错误，而是为了保持人类对系统输出的感知力。

## 十二、从零开始的第一步

说到底，Loop Engineering 并没有那么神秘。它做的，无非是把你一直在手工跑的那个循环，连同一条明确的验收标准，**交还给 Agent**。

今天就能开一个最小 Closed Loop 试起来。建议的顺序是：

1. **先 Closed，别一上来就放开边界**
2. **先小**：挑一件验收标准能写清楚的事
3. **先把"什么算达标"那条线划明白**

等这个小循环跑顺了，再考虑放开目标边界、加上 Orchestrator、接上定时触发，让它慢慢长成增长循环的样子。

逐步放开的路径：

```
Closed Loop（补测试）→ 加 Orchestrator（多 Agent 并行）→ 加 Scheduling（定时触发）→ 逐步放开目标边界 → Open Loop（自主探索）
```

**这件事的乐趣或许就在于：你亲手把自己从循环里请了出来，又看着它替你转了起来。**
