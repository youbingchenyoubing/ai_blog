# 逆向工程实战

> 逆向的本质只有一件事：**把编译器做的事逆回去**。编译器把人的逻辑变成机器指令，逆向就是从机器指令还原人的逻辑。本文不讲工具操作手册，只讲逆向的思维方法和比赛中的实战套路。

---

## 一、逆向的本质

### 为什么能逆向

```
源代码 → 编译器 → 机器码

编译是确定性的映射：同一段源码，同一编译器，同一优化级别，产出相同的机器码。
逆向利用的就是这个确定性——机器码中保留了源码的逻辑结构，只是表达形式变了。
```

**逆向做不到完美还原源码（变量名、注释丢失），但能还原程序的行为逻辑——这对CTF足够了。**

### 逆向的核心循环

```
观察现象 → 定位关键代码 → 理解逻辑 → 构造输入

1. 观察现象：程序做了什么？输入什么？输出什么？
2. 定位关键代码：找到判断flag的逻辑在哪
3. 理解逻辑：这个判断条件是什么？算法是什么？
4. 构造输入：根据逻辑算出正确的输入（flag）
```

## 二、前置知识

### 2.1 汇编——逆向的"英语"

不需要会写汇编，但必须能读。逆向中99%的时间在读汇编，只有1%在写。

**x86-64 核心指令（掌握这些就能读大部分代码）：**

| 类别 | 指令 | 含义 |
|------|------|------|
| 数据移动 | `mov dst, src` | dst = src |
| 数据移动 | `lea dst, [addr]` | 加载有效地址（不访问内存），lea rax, [rbp-0x10] 取栈地址 |
| 数据移动 | `movsx/movzx dst, src` | 带符号扩展/零扩展移动，处理 int→long 等类型转换 |
| 数据交换 | `xchg a, b` | 交换a和b的值 |
| 算术运算 | `add/sub/imul/idiv` | 加/减/乘/除 |
| 算术运算 | `inc/dec/neg` | 自增1/自减1/取负 |
| 算术运算 | `adc/sbb` | 带进位加/带借位减（实现大整数运算） |
| 自增比较 | `cmpxchg dst, src` | 比较并交换（原子操作），dst==rax则dst=src，否则rax=dst |
| 逻辑运算 | `and/or/xor/not/shl/shr` | 与/或/异或/非/左移/右移 |
| 逻辑运算 | `sar` | 算术右移（保留符号位），与shr区分 |
| 比较 | `cmp a, b` | 计算a-b设置标志位 |
| 测试 | `test a, b` | 计算a&b设置标志位 |
| 跳转 | `je/jne/jg/jl/jge/jle` | 等于/不等于/大于/小于/大于等于/小于等于时跳转 |
| 跳转 | `ja/jb/jae/jbe` | 无符号比较：高于/低于/高于等于/低于等于时跳转 |
| 跳转 | `js/jns/jo/jno` | 负/非负/溢出/无溢出时跳转 |
| 跳转 | `jmp addr` | 无条件跳转 |
| 条件传送 | `cmove/cmovg/cmovl...` | 条件成立时传送（避免分支，优化器常用） |
| 函数调用 | `call/ret` | 调用函数/返回 |
| 函数调用 | `leave` | 等价于 mov rsp, rbp; pop rbp（函数尾声） |
| 栈操作 | `push/pop` | 入栈/出栈 |
| 栈操作 | `pushad/popad` | 一次性入/出栈所有通用寄存器（x86壳中常见） |
| 字符串操作 | `movsb/stosb/rep` | 批量内存拷贝/填充，rep前缀配合rcx计数 |
| 取反 | `not dst` | 按位取反（~dst），区别于neg（-dst） |
| 空操作 | `nop` | 占位无操作，常用于patch填充 |
| 系统调用 | `syscall` | Linux系统调用（x86-64），配合rax指定调用号 |
| 系统调用 | `int 0x80` | Linux系统调用（x86 32位） |
| 断点 | `int 3` / 字节 `0xCC` | 软件断点指令，也被反调试用来检测 |

**关键寄存器：**

```
通用寄存器（x86-64）：
  rax — 函数返回值
  rdi, rsi, rdx, rcx, r8, r9 — 函数参数（按顺序）
  rbx, r12-r15 — 被调用者保存（函数内用前必须保存）
  rsp — 栈顶指针
  rbp — 栈底指针（帧指针）

标志寄存器关键位：
  ZF — 零标志（结果为0时置1）
  SF — 符号标志（结果为负时置1）
  CF — 进位标志
  OF — 溢出标志
```

**逆向中常见的汇编语句模式（识别这些能极大提速读码）：**

```
xor eax, eax            → 把eax清零（比mov eax,0更短，编译器优化首选）
test eax, eax           → 检查eax是否为0（常用于if(!x)，等价于cmp eax, 0）
cmp dword [rbp-4], 0    → 判断局部变量是否为0
je  short loc_401234    → 为0跳转（配合上面的cmp/test）
movzx eax, byte [rbp-4] → 读取单字节并零扩展到int（字符处理常见）
movsxd rax, eax         → 32位符号扩展到64位（int转long）
lea rax, [rbp-0x10]     → 取局部变量地址（常作为函数参数传入）
lea rax, [rdi+rdi*2]    → rax = rdi*3（编译器用lea代替乘法，更快）
shl eax, 4              → eax << 4（等价于乘以16）
sar eax, 31             → 算术右移31位（提取符号位：0或-1）
imul eax, 9             → eax *= 9（三操作数乘法）
cdq / cqo               → 符号扩展edx:rax（idiv前的准备）
idiv ecx                → eax=商，edx=余数（除法后取商取余）
mov eax, 0x12345678     → 加载32位立即数
movabs rax, 0x...       → 加载64位立即数（地址/大常量）
and eax, 0x7FFFFFFF     → 清除最高位（取正数/掩码）
or eax, 1               → 最低位置1（奇偶判断、标志位设置）
xor reg, reg            → 寄存器自身异或=清零（识别"置0"的标志）
push rbp; mov rbp, rsp  → 函数序言（建立栈帧）
mov rsp, rbp; pop rbp   → 函数尾声（等价于 leave）
sub rsp, 0x20           → 在栈上分配局部变量空间
call sub_401234         → 调用函数（IDA下划线前缀=未命名）
jmp short loc_401234    → 无条件跳转（常对应goto/continue/break）
mov [rip+0x2000], eax   → RIP相对寻址（位置无关代码PIC，Linux默认）
rep movsb               → 批量拷贝字符串（rcx=长度，rsi=源，rdi=目的）
lock cmpxchg [rsp], rax → 原子操作（多线程同步常见）
nop                     → 空指令（对齐/patch填充）

# —— 高级语言结构对应的汇编模式 ——

# 数组访问 a[i]（i为变量下标）
mov eax, [rbx+rcx*4]    → int数组：基址+下标*4
movzx eax, [rbx+rcx]    → char数组：基址+下标*1
mov rax, [rbx+rcx*8]    → 指针数组：基址+下标*8

# 二维数组 a[i][j]（列数为N）
mov eax, [rbx+rsi*4*N+rdx*4]  → 行优先，row*i + col

# 结构体访问 obj.field（field偏移0x10）
mov rax, [rbx+0x10]     → rbx=对象指针，0x10=字段偏移
mov [rbx+0x10], rax     → 写入字段

# for循环（i=0; i<n; i++）
mov [rbp-4], 0          → i=0
loc_loop:
cmp [rbp-4], n
jge loc_end             → i>=n退出
... 循环体 ...
mov eax, [rbp-4]
add eax, 1
mov [rbp-4], eax        → i++
jmp loc_loop
loc_end:

# while循环
loc_while:
... 判断条件 ...
jz loc_end              → 条件为假跳出
... 循环体 ...
jmp loc_while
loc_end:

# if-else
cmp eax, 0
jne loc_else            → 条件为假跳到else
... if分支 ...
jmp loc_end
loc_else:
... else分支 ...
loc_end:

# switch（少量case用cmp链，多case用跳表）
cmp eax, 0
je  case_0
cmp eax, 1
je  case_1
jmp default
# 跳表形式：jmp [table + eax*8]，IDA中显示为switch跳表

# 函数调用（传递参数）
mov edi, 100            → 第1个参数
mov esi, 200            → 第2个参数
xor eax, eax           → 变参函数需清零AL（告诉libc无浮点参数）
call printf

# 函数返回
mov eax, 0x1234         → 设置返回值
ret                     → 返回到call的下一条指令

# 栈保护（Canary，防栈溢出）
mov rax, fs:[0x28]      → 读取canary
mov [rbp-8], rax        → 存到栈上
... 函数体 ...
mov rax, [rbp-8]
xor rax, fs:[0x28]      → 检查是否被破坏
jne stack_check_fail    → 不一致则调用__stack_chk_fail
leave; ret

# 浮点运算（SSE2）
cvtsi2sd xmm0, eax      → int转double
addsd xmm0, xmm1        → double加法
mulsd xmm0, xmm1        → double乘法
comisd xmm0, xmm1       → double比较

# 取模运算（编译器优化）
mov eax, edx            → idiv后edx是余数，相当于 x % n
# 模2的幂：and eax, (n-1)（编译器必优化）
and eax, 7              → x % 8

# 乘法优化为移位与加减（编译器常见）
lea eax, [rdi+rdi*4]    → x*5
shl eax, 3              → x*8
lea eax, [rdi*8+rdi]    → x*9
lea eax, [rdi+rdi*2]
shl eax, 3              → x*24
```

### 2.2 调用约定

```
Linux (System V AMD64 ABI)：
  参数传递：rdi → rsi → rdx → rcx → r8 → r9 → 栈
  返回值：rax
  栈对齐：call前RSP必须是16的倍数

Windows (x64)：
  参数传递：rcx → rdx → r8 → r9 → 栈
  返回值：rax
  影子空间：调用者预留32字节

识别调用约定的作用：
  看到mov rdi, xxx → call → Linux约定
  看到mov rcx, xxx → call → Windows约定
  由此判断参数个数和类型
```

### 2.3 可执行文件格式

```
ELF (Linux)：
  ELF Header → 文件类型、入口点、架构
  Program Headers → 内存段布局（加载器用）
  Section Headers → 代码段(.text)、数据段(.data/.bss)、符号表(.symtab)

PE (Windows)：
  DOS Header → PE签名
  Optional Header → 入口点、镜像基址
  Section Table → .text/.data/.rdata/.rsrc

逆向关注点：
  入口点（Entry Point）→ 程序从哪开始执行
  .text段大小 → 代码量，判断是否有壳
  .symtab是否存在 → 有符号表则函数名可读，逆向难度骤降
```

## 三、静态分析

### 3.1 思路

静态分析的核心是**不运行程序，通过阅读反编译代码理解逻辑**。

```
IDA Pro / Ghidra 工作流：

1. 找入口点 → F5反编译main
2. 识别关键函数 → 看字符串引用（Shift+F12）→ 交叉引用（X键）定位
3. 读伪代码 → 理解算法逻辑
4. 标注 → 重命名变量/函数，加注释，让逻辑越来越清晰
```

### 3.2 定位关键代码

**最快的方法：字符串引用。**

```
程序总要输出"correct"/"wrong"/"flag{"之类的字符串
  → IDA中 Shift+F12 打开字符串窗口
  → 找到关键字符串
  → 双击 → 交叉引用(X) → 直接跳到判断逻辑

没有明显字符串？
  → 看输入函数：scanf/gets/read/fgets → 交叉引用找调用点
  → 看比较函数：strcmp/memcmp/strncmp → 交叉引用找比较逻辑
  → 看加密函数：AES/DES/RC4/TEA/XTEA 的特征常量
```

### 3.3 识别常见算法

**加密算法的特征常量——逆向中的"指纹"：**

| 算法 | 特征常量/特征操作 |
|------|-------------------|
| AES | S盒 `0x63,0x7c,0x77,0x7b...`，轮常量 `0x01,0x02,0x04...` |
| DES | 初始置换表，S盒，P盒 |
| RC4 | 初始化循环 `for(i=0;i<256;i++) S[i]=i`，swap操作 |
| TEA/XTEA | 魔数 `0x9E3779B9`（delta），循环32次，`+=`/`^=`交替 |
| MD5 | 初始值 `0x67452301,0xEFCDAB89...`，64轮循环 |
| SHA1 | 初始值 `0x67452301,0xEFCDAB89,0x98BADCFE...`，80轮 |
| SHA256 | 初始值 `0x6a09e667...`，64轮，K常量表 |
| Base64 | 字符表 `ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/` |

**看到0x9E3779B9就是TEA/XTEA，看到0x67452301就是MD5/SHA——这是逆向的速判技巧。**

### 3.4 读反编译代码的方法

```
核心原则：不要逐行读，先抓结构再填细节

1. 识别控制流
   if-else → 看cmp/test后的分支
   循环 → 看跳回指令，判断是for/while
   switch → 跳表结构

2. 识别数据流
   输入在哪？→ 追踪输入变量的使用
   比较在哪？→ 找到flag判断点
   中间做了什么？→ 加密/编码/变换

3. 识别函数功能
   看参数和返回值 → 推断函数用途
   看内部操作 → 确认推断
   重命名函数 → 让代码可读
```

## 四、动态调试

### 4.1 什么时候需要动态调试

```
静态分析够用时不需要动态调试。以下情况必须动态：

1. 反编译代码看不懂 → 运行时观察变量值
2. 有反调试 → 需要绕过
3. 有自解密/加壳 → 需要脱壳后dump
4. 算法复杂 → 需要单步跟踪确认理解
5. 需要修改跳转 → 直接patch掉判断
```

### 4.2 GDB（Linux）

```bash
# 启动
gdb ./program
gdb -q ./program  # 安静模式

# 断点
b main              # 在main下断
b *0x401234         # 在地址下断
b func_name         # 在函数下断
b file.c:42         # 在源码行下断
delete              # 删所有断点

# 执行
r                   # 运行
r arg1 arg2         # 带参数运行
c                   # 继续执行
si                  # 单步进入（进函数）
ni                  # 单步越过（不进函数）
finish              # 执行到当前函数返回

# 观察
info registers      # 查看所有寄存器
p $rax              # 打印寄存器
p/x $rax            # 十六进制打印
p/s (char*)$rdi     # 当字符串打印
x/20wx $rsp         # 查看栈内存（20个4字节十六进制）
x/s 0x404000        # 查看字符串

# 修改
set $rax = 0        # 修改寄存器
set {int}0x404000 = 42  # 修改内存

# pwndbg 增强插件（推荐）
# 安装后自动显示上下文：寄存器、栈、反汇编、回溯
```

### 4.3 x64dbg（Windows）

```
基本操作（与GDB对应）：

断点：F2（下断） / F9（运行） / F7（单步进入） / F8（单步越过）
观察：寄存器窗口 / 内存窗口 / 栈窗口
修改：右键寄存器→修改 / 内存窗口Ctrl+E编辑
补丁：右键→Assemble修改指令 → Patches窗口导出
```

### 4.4 调试策略

**CTF逆向最常用的调试手法：**

```
1. 在比较点下断 → 观察比较的两个值
   找到strcmp/memcmp调用 → 断点 → 看参数 → 一个是输入，一个是正确答案

2. 修改跳转 → 绕过判断直接拿flag
   je → jne（或反过来） → 程序输出正确结果
   x64dbg中右键→Assemble改指令
   GDB中 set {char}0x地址 = 0x75 (je→jne)

3. 追踪加密过程 → 记录中间值
   在加密函数入口/出口下断 → 观察输入输出 → 理解变换

4. dump内存 → 获取解密后的代码/数据
   自解密程序 → 在解密完成后断下 → dump .text段
```

## 五、CTF逆向常见题型

### 5.1 算法还原

**最基础也是最常见的题型：程序实现了一个算法，要求还原算法并求出正确输入。**

```
解题流程：
1. 找到输入点（scanf/gets/read）
2. 追踪输入数据的变换过程
3. 找到比较点（strcmp/memcmp/逐字符比较）
4. 逆推算法，从比较目标反推输入

常见变换：
  逐字符变换 → XOR/加减/查表 → 逐个逆推即可
  整体变换 → 加密算法 → 识别算法后用解密函数
  自定义变换 → 仔细分析，写逆运算脚本
```

**XOR加密的识别与还原：**

```c
// 正向：加密
for(i = 0; i < len; i++)
    cipher[i] = plain[i] ^ key[i % key_len];

// 逆向：解密（XOR是自逆运算）
for(i = 0; i < len; i++)
    plain[i] = cipher[i] ^ key[i % key_len];

// 识别特征：单条指令 xor reg, reg 或 xor [mem], reg
// 如果key是单字节：cipher ^ key = plain，直接异或回去
```

**TEA加密的识别与还原：**

```c
// TEA加密（看到0x9E3779B9就是它）
void encrypt(uint32_t *v, uint32_t *key) {
    uint32_t v0=v[0], v1=v[1], sum=0, delta=0x9E3779B9;
    for(int i=0; i<32; i++) {
        sum += delta;
        v0 += ((v1<<4)+key[0]) ^ (v1+sum) ^ ((v1>>5)+key[1]);
        v1 += ((v0<<4)+key[2]) ^ (v0+sum) ^ ((v0>>5)+key[3]);
    }
    v[0]=v0; v[1]=v1;
}

// TEA解密——把加密的+=改成-=，循环倒过来
void decrypt(uint32_t *v, uint32_t *key) {
    uint32_t v0=v[0], v1=v[1], sum=0xC6EF3720, delta=0x9E3779B9;
    // sum = delta * 32
    for(int i=0; i<32; i++) {
        v1 -= ((v0<<4)+key[2]) ^ (v0+sum) ^ ((v0>>5)+key[3]);
        v0 -= ((v1<<4)+key[0]) ^ (v1+sum) ^ ((v1>>5)+key[1]);
        sum -= delta;
    }
    v[0]=v0; v[1]=v1;
}
```

### 5.2 反调试

**程序检测自己是否被调试，检测到则改变行为。**

```
常见反调试手段及绕过：

1. ptrace检测（Linux）
   if(ptrace(PTRACE_TRACEME, 0, 1, 0) < 0) exit();
   // 原理：一个进程只能被一个调试器attach
   // 绕过：patch掉ptrace调用，或hook ptrace返回0

2. IsDebuggerPresent（Windows）
   if(IsDebuggerPresent()) exit();
   // 绕过：修改返回值为0，或patch跳转

3. 检测调试端口
   读取 /proc/self/status 看TracerPid字段
   // 绕过：hook read/fopen，返回伪造内容

4. 时间检测
   start = time(); ... ; if(time()-start > threshold) exit();
   // 绕过：hook time/gettimeofday返回固定值

5. int 3 断点检测
   在关键代码前检查是否有0xCC（int3指令）
   // 绂过：用硬件断点代替软件断点（GDB: hbreak）

通用绕过思路：
  找到检测代码 → patch掉（NOP填充）或修改跳转
  找不到检测代码 → 在程序入口最早处下断，单步跟踪定位
```

### 5.3 加壳与脱壳

**壳的本质：给程序加了一层解密stub，运行时先解密再跳转执行原程序。**

```
识别加壳：
  1. DIE(Detect It Easy) / ExeinfoPE 查壳
  2. 入口点不在正常的.text段
  3. IDA中看到大量call/jmp但逻辑混乱
  4. 节区名异常（UPX0/UPX1/.vmp/.nsp0等）

常见壳及脱壳方法：

UPX（最常见，CTF常考）：
  upx -d packed_file -o unpacked_file  # 一行命令脱壳
  如果-d被改了标志位 → 修复UPX头部标志后重试

VMProtect / Themida（高级壳）：
  动态脱壳：运行到OEP(原始入口点) → dump内存 → 修复IAT
  找OEP的方法：
    - 单步跟踪，等解密循环结束
    - 在VirtualProtect下断（壳解密后要改内存属性）
    - 看ESP变化（pushad/popad法）

自写壳（CTF常见）：
  1. 找到解密循环 → 在循环结束后下断
  2. dump内存中的代码段
  3. 修复入口点和节区属性
```

### 5.4 Android逆向

```
APK结构：
  classes.dex → Dalvik字节码（主要分析目标）
  resources.arsc → 资源
  lib/ → native库（.so文件）
  AndroidManifest.xml → 权限和组件声明

工具链：
  jadx → 反编译DEX为Java源码（首选）
  apktool → 解包APK，修改资源，重打包
  Frida → 运行时hook（动态分析利器）

  3. 有混淆 → Frida hook绕过或动态追踪

Frida速查：
```

## 六、实战解题方法论

### 6.1 通用解题流程

```
Step 1：信息收集（2分钟）
  file命令 → 文件类型和架构
  DIE/ExeinfoPE → 是否加壳
  strings → 关键字符串（flag/密码/URL/算法名）
  运行程序 → 看输入输出行为

Step 2：静态分析（10分钟）
  IDA/Ghidra打开 → F5反编译main
  字符串引用 → 定位关键判断逻辑
  读伪代码 → 理解算法
  识别加密算法 → 查特征常量

Step 3：动态验证（按需）
  关键点下断 → 验证静态分析的理解
  观察变量值 → 确认算法参数
  修改跳转 → 快速验证猜想

Step 4：求解
  算法可逆 → 写解密脚本
  算法不可逆 → 爆破（z3约束求解/angr符号执行）
  比较点已知 → 直接提取目标值
```

### 6.2 Z3 约束求解

**当算法复杂但逻辑清晰时，用Z3自动求解比手动逆推更高效。**

```python
from z3 import *

s = Solver()
flag = [BitVec(f'f{i}', 8) for i in range(32)]  # 32字节flag

# 约束：可打印字符
for c in flag:
    s.add(c >= 0x20, c <= 0x7e)

# 从反编译代码中提取的约束条件
s.add(flag[0] + flag[1] == 0xAB)
s.add(flag[2] ^ flag[3] == 0x55)
# ... 逐条添加

if s.check() == sat:
    m = s.model()
    result = ''.join(chr(m[f].as_long()) for f in flag)
    print(result)
```

### 6.3 angr 符号执行

**当不想手动分析算法时，让angr自动探索路径。**

```python
import angr

proj = angr.Project('./program', auto_load_libs=False)

# 从输入点开始符号执行
state = proj.factory.entry_state(stdin=angr.SimFile('/dev/stdin', size=64))

# 定义找到和避免的地址
find_addr = 0x401234   # "correct"分支的地址
avoid_addr = 0x401256  # "wrong"分支的地址

simgr = proj.factory.simulation_manager(state)
simgr.explore(find=find_addr, avoid=avoid_addr)

if simgr.found:
    found = simgr.found[0]
    print(found.posix.dumps(0))  # 打印满足条件的输入
```

### 6.4 常见坑点

```
1. 整数溢出
   比较用有符号，计算用无符号 → 结果非预期
   识别：看是否有movsx/movzx（符号扩展/零扩展）

2. 大端/小端
   x86是小端（低字节在低地址）
   网络字节序是大端
   逆向时注意字节序转换（ntohl/htonl）

3. 结构体对齐
   IDA可能把结构体成员识别为独立变量
   手动创建结构体 → 让反编译更清晰

4. 优化导致的代码变形
   -O2优化后循环可能被展开
   乘除可能被替换为移位和加减
   识别：乘以3 → lea rax, [rdi+rdi*2]；除以常量 → 乘以魔数再右移
```

### 7.2 静态分析：快速理解代码逻辑

**核心原则：不要逐行读，先建立"函数地图"再切入关键逻辑。**

#### 步骤1：定位关键代码（最快路径——字符串）

```
1. Shift+F12 打开字符串窗口
2. 搜索关键词："correct"、"wrong"、"flag"、"password"、"error"
3. 双击字符串 → 跳到 .rodata 段该字符串位置
4. 按 X（Cross-References to）→ 看到所有引用该字符串的位置
5. 双击引用 → 直接跳到判断逻辑所在的函数
```

没有明显字符串时：

```
- 看输入函数：scanf/gets/fgets/read → Shift+F12搜"scanf"或Import表查找
- 看比较函数：strcmp/memcmp/strncmp → Import表→找函数→X引用
- 看加密常量：TEA的0x9E3779B9 / MD5的0x67452301 → 搜常量值
  方法：Search → Sequence of bytes / Immediate value
```

#### 步骤2：阅读伪代码（F5）

在函数内按 `F5` 切换到 Hex-Rays 伪代码视图，逻辑会瞬间清晰。

**让伪代码更可读的三大操作：**

```
1. 重命名变量/函数（N）
   - 选中 v1 → 按 N → 输入 input_len
   - 选中 sub_401234 → 按 N → 输入 check_flag
   - 重要：边读边改名，逻辑会自然浮现

2. 修改变量类型（Y）
   - 选中 _BYTE* v2 → 按 Y → 改为 char* v2
   - 选中 int v3 → 按 Y → 改为 struct Student* v3
   - 改完后伪代码立即重渲染：v3[12] → v3->age

3. 结构体定义（Shift+F9 → Insert）
   - View → Open subviews → Structures
   - Insert 新建结构体 → 逐一添加字段
   - 在伪代码中按 Y 应用结构体类型
```

#### 步骤3：标记与注释

```
- 注释（; 或 /）：在反汇编行加注释 / 伪代码行加注释（/）
- 函数注释：选中函数名 → 右键 → Function comment
- 重复注释（Ins）：跨所有引用显示的注释
- 颜色标注：右键 → Color → 给关键函数/变量加颜色
```

#### 步骤4：理解控制流

```
- Ctrl+Alt+F：Function Calls 窗口 → 看函数调用层次
- 选中函数 → Ctrl+X：双向交叉引用（Xref to/Xref from）
- 图形视图：直观显示跳转关系（绿色=条件成立，红色=不成立，蓝色=无条件）
- 选中 var → 选中 var → 按 = 键高亮所有引用（或Ctrl+L看列表）
```

### 7.3 自带调试器：完整动态调试流程

IDA 自带调试器，可下断点、单步、改寄存器、修改内存，且能与静态伪代码无缝切换——这是它相对 GDB/x64dbg 的最大优势。

#### 1. 选择调试器并启动

```
菜单：Debugger → Select debugger
  - Local Windows debugger   → 调试本地 .exe
  - Local Linux debugger     → 调试本地 ELF
  - Remote GDB debugger      → 调试远程 Linux/嵌入式设备
  - WinDbg                  → 调试 Windows 内核/特殊场景

启动调试：
  Debugger → Start Process（快捷键 F9）
  - 若需要在入口前下断：勾选 "Suspend on library load/unload"
```

#### 2. 下断点

```
F2      → 在当前反汇编/伪代码行下断点（最常用）
右键    → Add Breakpoint（指定地址）
F9      → 运行到下一个断点

常用下断位置：
  - main 入口（先用 G 跳到 main 地址 → F2）
  - strcmp/memcmp/strncmp 调用处
  - cmp 指令前（验证逻辑点）
  - 加密函数入口
```

#### 3. 执行控制快捷键

| 快捷键 | 含义 | 说明 |
|--------|------|------|
| `F9` | Run | 运行到下一断点 |
| `F7` | Step Into | 单步进入（钻入函数） |
| `F8` | Step Over | 单步越过（不进函数） |
| `Ctrl+F7` | Step Out | 运行到当前函数返回 |
| `F4` | Run to Cursor | 运行到光标处 |
| `Ctrl+F2` | Terminate Process | 终止并重启进程 |
| `Alt+F2` | Suspend | 暂停进程 |

**最大优势**：在伪代码视图（F5）中也能直接 `F2` 下断、`F8/F7` 单步，IDA 会同步高亮当前执行的伪代码行和反汇编指令，逻辑清晰。

#### 4. 观察运行状态

调试时打开以下窗口（`Debugger` → `Open debugger windows`）：

```
- Registers  查看寄存器值（红色=刚修改过的，一眼看出变化）
- Stack      查看栈内容和函数返回地址
- Hex View   查看任意地址的内存（Ctrl+G 输入地址跳转）
- Locals     查看当前函数局部变量
- Watch      添加监视表达式
- Modules    查看已加载的模块/DLL/SO，可双击查看其导出函数
- Threads     查看线程（多线程程序调试用）
```

#### 5. 修改寄存器与内存（逆向核心操作）

```
改寄存器值：
  Registers 窗口 → 双击寄存器 → 输入新值（支持十进制/十六进制）

改内存：
  Hex View → 选中字节 → F2 编辑 → 编辑完 Ctrl+W 保存

修改后立即生效：
  F9 继续执行 → 程序感知的是修改后的值
```

#### 6. 修改跳转指令（CTF 逆向最常用技巧）

绕过 flag 验证的两种方法：

```
方法A：运行时改寄存器/标志位（不修改代码）
  1. 在 cmp eax, 0 处 F2 下断 → F9 运行
  2. 命中后，Registers 窗口改 ZF=1（或改 cmp 两操作数相同）
  3. F8 步过 → 跳转按预期执行 → 程序输出 "correct"

方法B：直接 Patch 指令（永久修改）
  1. 右键 je 指令 → Patch program → Assemble
  2. 输入 jne（机器码 0x74 改 0x75）
  3. Edit → Patch program → Apply patches to input file → 保存为新 exe
  4. 直接运行新 exe → 永久绕过验证
```

#### 7. 完整实战流程示例

```
题目：一段加密后比较 flag 的程序

1. File → Open → 选 crackme.exe → 默认 Load → 等待分析完成
2. Shift+F12 → 搜 "wrong" → X → 跳到判断函数 check
3. F5 反编译 check 函数 → 看到伪代码：
     if ( strcmp(encrypt(input), target) == 0 )
         puts("correct");
     else
         puts("wrong");
4. 在 strcmp 调用行 → F2 下断
5. Debugger → Select debugger → Local Windows debugger
6. Debugger → Start Process (F9) → 程序启动
7. 在程序窗口输入测试字符串 "AAAA" 回车
8. 命中 strcmp 断点 → 看 rdx 寄存器 (或 Locals 窗口)
   → 发现 target = "real_flag_str" → 直接拿到结果
9. 若需要绕过：右键 je → Assemble → 改 jne → F9 → 程序输出 "correct"
10. 保存补丁：Edit → Patch program → Apply patches → 输出 crackme_patched.exe
```

### 7.7 常见问题与坑点

```
1. F5反编译失败？
   → 函数识别错误：右键 → Force Analyze
   → 非函数代码段：P 标记为函数后 F5
   → 栈不平衡：Options → General → Disassembly → 勾选"Stack pointer"
     → 找到红色（不平衡）行 → 手动修正

2. 伪代码中显示不出结构体？
   → Shift+F9 新建结构体 → 在伪代码中按Y应用类型

3. 调试时无法命中断点？
   → ASLR（地址随机化）：Debugger → Debugger options → 取消勾选ASLR
   → 程序检测调试器：先 patch 掉反调试代码

4. idb 数据库越来越大怎么办？
   → File → Save → 保存到新的 idb（旧库可保留作备份）

5. IDA卡死/分析超慢？
   → 选项 → General → Analysis → 调整启发式级别
   → 大固件：分块分析
```
